spring beans相关

1.BeanFactory: 管理Bean的工厂，负责Bean的创建、依赖关系处理。
2.BeanDefinition: 存放Bean对应的class信息，包括当前类是不是抽象类，类的成员变量之类的。主要实现类有RootBeanDefinition和ChildBeanDefinition。
3.BeanDefinitionRegistry: 存放Bean的书架，负责Bean的注册。
4.BeanListableFactory: 实现了BeanFactory和BeanDefinitionRegistry两个接口

lombok工具

lombok可以消除java代码中一大堆的getter，setter，toString,hashCode,equal等默认代码，也能避免代码覆盖工具对这些代码的检测。
实现原理是lombok在java代码编译的时候，修改语法树(AST)增加getter和setter节点

  • idea中装lombok plugin
    注意1：安装需要重启idea
    注意2: maven编译报缺少getter方法，需要把maven-compiler-plugin升级到3.3（pom文件）

杂项

互金风控杂想

在一家互联网金融公司呆了两年左右时间，记录一些感想：

  1. 互联网金融是个新兴行业，它以后肯定不会消亡，但是行业规模不会大，因为中国金融体系基本上国有控股，银行占据核心地位，而且互联网金融很容易受到国家监管。
  2. 跟其它行业类型，互联网金融也会有8-2或者7-3-1这样的分布方式，某几个巨头会占据80%的业务，这样风控就显得比较尴尬了。因为大公司会有自己的风控，不会用第三方风控，小公司业务量太少，也不会太多的利润贡献给风控公司。未来两年P2P公司肯定会兼并融合，风控公司还要种模式就是和银行或者保险公司合作，赚它们的钱。
  3. 业务离钱越近越容易赚钱，例如信贷类业务直接关系钱，它的利润就比其他视频安全，内容安全等反欺诈业务高很多，很简单，视频网站和论坛网站自己的利润就那么点，怎么会有很多利润贡献给风控公司呢？
  4. 如果单纯的做信贷业务，其实不需要太复杂的技术和机器成本投入，因为业务相对固定而且量少，另外信贷的业务数据是相对比较独立的。但是如果要做内容安全或者其它反欺诈，量太多，这个就比较考验技术了。但是投入性价比其实很低的。
  5. 风控行业到后面还是靠信贷类产品，融资什么的，调用量基本上作用不大，投资者更关注的是公司的业务模式和盈利情况。这好比小龙虾市场价16块钱1斤，我烧好后卖20块，其它人卖50块。那我肯定销量第一，但是不赚钱，模式不可持续。
  6. P2P的坏账率和逾期率其实很高，银行低是因为银行准入条件太严格。而互金行业准入条件太低，而且借贷者违约成本太低。我暗地追踪了几个专门褥羊毛的(各P2P平台专门借钱，但是恶意违约的)，发现只有多头借贷是个很好的风控技术，其它的效果甚微。 一旦借贷者违约，很多贷后、逾期工具效果也不好。因为P2P公司追账成本太高，别人就借了1-2K，不停电话，各种短信，人力成本很高，而且还冒着法律的风险，更关键的是，恶意违约的人根本不care这些。有些P2P倒是想出了一个歪点子，直接读取借贷者的通讯录，不还就通讯录一路电话和短信过去，这样有些借贷者会迫于亲戚朋友压力，也会还些。
  7. 目前，感觉数据类产品的风控效果会比模型类产品效果要好些，例如上面提到的多头借贷数据。很简单，中国缺少信用体制，很多人是没有信用记录的，它的日常生活也很少产生信用数据。模型缺少有效的数据支撑，效果自然不会好。例如，基于淘宝交易主维度的芝麻分之类的产品肯定在信贷上效果不好。我追踪的那几个褥羊毛的人，看他们的交易记录跟普通人完全没区别的。还有就是模型的逻辑无法反应真实情况，例如，褥羊毛或者团伙欺诈，想通过模型识别，基本上不可能。

预测楼市涨的一个有效指标

2016年4月时候，杭州楼市还处于不火的时候，有个亲戚想在杭州买房，问我杭州房价会不会涨，于是我想搞个模型来预测下杭州的房价
那个正好接触到ELK，想熟悉下这套框架，其实搭建这东西很简单的，就10分钟的事情。后面花了个周末写了一个爬虫系统，爬取了透明网的房价数据。
后面就是重点了，用什么指标来判断房价走势，常用的无非这几种:
(1)成交均价，如果按季度或者更长单位来看，均价是有效的，但是如果以日或者月来计算，那么这个指标是不敏感的，比如，3月份开的楼幢是楼王，它的成交均价肯定比4月份开的炮灰楼幢高很多。
所以，短时间的均价不能反映楼盘涨跌趋势。兔博士实际上就是用这个方式的，也是很多机构采用的方式。
(2)成交量，同样问题，受制于开发商每个月的开放楼盘数量差异，而且每天卖的房子数量也不均一，所以成交量也不能反映楼盘涨跌趋势。
(3)基于上面分析，基于成交均价和成交量的任何权重模型无法反映短时间内楼盘的涨跌

春江水暖鸭先知，开发商肯定是最了解房价涨跌的人。如果有个指标能体现开发商的行为，那最合适不过了。我发现还真有。

这个指标就是：同个房间的成交价/最初这个房间的备案价。

目前的现状是透明网有每个房间的备案价，但是没有这个房间的成交价。大概花了1个周末，用相对复杂的逻辑(组合、排列、预测、修正)解决掉了这个问题，得到每个房间的成交价。
很多爬过透明网的人都问我怎么知道每个房间的成交价，以为我黑了透明网的数据库。其实不是的，我最初得到的数据跟他们是一样的。只不过最终我解决了。

来说说为什么这个指标在房价上涨的时候效果是非常灵敏的。讲下卖房子流程，开发商要卖房子之前，需要在房管局备案每个房间的价格，这个价格包含了开发商对整个房地产市场的预期和对这个房间的预期。房地产市场好，这个价格可能就高，注意：房管局规定成交价是不能高于备案价的。为什么一定要“最初这个房间备案价”呢，很简单，备案价是可以调整的，例如开发商发现市场转热，它可以上调备案价。所以，只要这个指标逐步上升，说明开发商给客户的折扣就越少，房地产越火热。

大概在2016年6-7月时候，我发现杭州出现了第一个 “同个房间的成交价/最初这个房间的备案价”》1的记录，记得是个申花板块的某个楼盘。这是什么情况呢？大概6月爬取这个房间的“最初的备案价”是32000元，然后成交价是34000，成交价怎么可能大于备案价呢？原因是7月开发商上调了备案价到35000，这样出现大于1的情况。这说明开发商已经有足够的信息能判断杭州2016年下半年楼市会涨，所以敢上调备案价。当时，我在杭州住房消息群里面说了：杭州房价即将暴涨了，但是大家都没留意，事后想想，这个时候入手无论自住还是投资都是一个非常好的时机。相差几个月，就是100W的事情，又要多干多少年。。。

后面的情况一目了然了，这个指标一直在上升，灵敏性非常好，甚至能看出，哪个开发商赚钱最狠，这个指标最高1.45，万科中央公园某个房间，万科果然是涨价最狠的开发商。。

进入11-12月，这个指标就失效了，每个楼盘的这个指标都趋近于1。很简单，楼市火热，开发商每开一幢，出个备案价，日光，所以成交价基本上等于备案价。

不过，这个时间已经无关紧要了。因为这个时候大家都认为杭州楼市会涨。
